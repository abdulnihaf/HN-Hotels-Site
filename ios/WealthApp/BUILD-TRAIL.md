## Architecture decisions (do NOT relitigate)
1. **Standalone native app**, NOT a Hukum chamber (owner boundary: keep separate from Chat 33 Voice / Chat 34 App).
2. **Logic stays server-side.** The app renders state + posts intent; it decides nothing.
3. **Auth = local PIN unlock + embedded DASHBOARD_KEY.** The PIN guards the surface on-device; the key authorizes the API. The key gates BOTH reads and order execution — treat it as order-capable.
4. **Mode banner fails UNSAFE.** If config can't be read → "MODE UNKNOWN", never "SAFE". Shows the live `block_real_orders` + `engine_mode` values. (Caught a false-"SAFE" bug by verifying against live config — config was the slowest of 6 reads; fix = load config first.)
5. **Orders forced to MIS** (intraday auto-square), never CNC. Idempotency tag `HN_WE_IOS_<ts>` + button disabled while placing.
6. **Practice mode default-on** so the flow can be rehearsed with zero risk / no Kite.

## The Connect-Kite saga (the hard-won lesson)
Goal: connect Kite without dumping the user on the web dashboard's key gate.
1. `openURL(/wealth/auth/login)` → opened **Chrome** (default browser), never handed off. ✗
2. Resolve `kite.zerodha.com/connect/login` + `UIApplication.open(.universalLinksOnly)` → **fell back to browser** (Zerodha does NOT register a universal link for third-party Connect logins). ✗
3. `SFSafariViewController` (in-app, shares Safari cookies) → cleaner, but still landed on `/trading/?kite=connected` which demands the dashboard key. ✗ (the user's complaint)
4. **FINAL: in-app Safari + status polling that auto-dismisses the sheet** the instant `/api/kite?action=status` reports `connected` (token stored server-side after Authorize). No backend deploy, no dashboard-key gate. ✓
- **LAW: third-party Kite Connect login is a WEB flow.** Zerodha only deep-links to the Kite app for its own first-party login. The Kite login page showing once is unavoidable.
- **"Invalid api_key" ≠ wrong key.** It means the Kite Connect app/subscription LAPSED. App `HN_Wealth_Engine` (api_key `85yfvys3t9t66l35`, redirect `https://trade.hnhotels.in/wealth/auth/callback`) was reactivated (extended to 26 Jul 2026) → SAME key works → our Cloudflare `KITE_API_KEY` already matched, no change needed.

## Build / install pipeline (proven)
- `xcodegen generate` after every `project.yml` / new-file change.
- Sim compile-check: `-sdk iphonesimulator -destination 'generic/platform=iOS Simulator' CODE_SIGNING_ALLOWED=NO`, DerivedData in **/tmp** (avoids synced-tree FinderInfo codesign trap).
- Device: `-sdk iphoneos -destination 'generic/platform=iOS' -allowProvisioningUpdates DEVELOPMENT_TEAM=FZ58DQ52QS` → `devicectl device install app` → `process launch`. (Use `generic/platform=iOS`, not the device id, when the phone drops to "unavailable" mid-session.)
