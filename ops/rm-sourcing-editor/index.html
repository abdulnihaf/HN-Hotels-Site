📖 Code Legend · Source of Truth

Decode any canonical code on this page without leaving.

Code Anatomy

{BRAND}-{TYPE}-{USAGE}-{SOURCING}-{ITEM}
BRANDHN / NCH / HE — outlet scope
TYPE2 chars: A/D + M/S
USAGEP / R / O + alts — production · retail · operational
SOURCING1-3 letters: L/B/I, case = primacy
ITEM3 chars, uppercase abbreviation

Example walkthrough

HN-AM-P-Lb-SUG
  • HN — cross-brand RM (used at both NCH and HE)
  • AM — approximate, multi-use
  • P — production only (consumed by our recipes, never resold)
  • Lb — primary loose, branded SKU also possible (in-house impossible — i absent)
  • SUG — Sugar

USAGE Profile · Position 3 · 1 to 3 characters

The three letters that can appear

Three-letter (primary + 2 alternates)

Pro, Por, Rpo, Rop, Opr, Orp — RM serves all three roles. The first letter is primary, the remaining two (lowercase) are alternates. Order of the alternates is owner's choice.

Forbidden: empty USAGE. An RM that's neither used, sold, nor consumed by operations has no operational reason to exist.

SOURCING Profile · Position 4 · 1 to 3 characters

The three letters that can appear

LetterMeaningReal example
LLoose — vendor-direct purchase, no defined SKUBuffalo milk from Prabhu @ ₹55/L
BBranded — defined SKU, specific brand × specific pack-sizeAmul 500g butter pouch
IIn-house — produced internally from upstream RMs (recipe-based)Bun if baked from maida + yeast + butter + sugar

Case convention — the key rule

UPPERCASE letter = currently primary (the way this RM is sourced today, most often) lowercase letter = alternate possible (also valid, but not primary) letter ABSENT = forbidden for this RM (operationally impossible)

Why absent letters mean "forbidden"

The system reads the sourcing slot to know which actions are valid for this RM.

  • HN-AM-P-Lb-SUG / I / R-… → REJECTED (I not in Lb, no in-house possible for sugar)
  • HN-AM-P-L-CHL / B / … → REJECTED (B not in L, no branded charcoal SKU)
  • HN-AM-O-B-LPG / L / … → REJECTED (L not in B, no loose LPG)

Enforced automatically — the editor validates at save time, the API rejects invalid combinations with 400 errors.

ITEM Abbreviation · Position 5 · 3 characters

RuleDescription
LengthExactly 3 characters, uppercase
SourceAuto-derived from item name (first 3 unique consonants)
OverrideOwner can manually override the auto-derived value
UniquenessMust be unique within (BRAND, TYPE) combination

Example derivations

  • "Sugar" → SUG
  • "Buffalo Milk" → MLK or BFM (owner picks)
  • "Osmania Biscuit" → OSB
  • "Almonds" → LMN

If the auto-derived abbreviation collides with an existing RM, the editor flags it and asks the owner to override.

Anomaly Guards · What the system catches automatically

Forbidden comboWhy it's caught
USAGE emptyAn RM must be at least one of P (production), R (retail), or O (operational). None = no operational reason to exist.
USAGE malformedFormat is 1 uppercase primary {P/R/O} + 0-2 lowercase alts {p/r/o}. Multiple uppercase (PR, PRO) rejected. Lowercase-first (pR) rejected.
USAGE duplicate letterPp, Oo, Rr rejected — each letter (P, R, O) appears at most once per slot.
Sourcing = I aloneAn RM must be purchasable. I-only items are states of production (decoction, malai), not RMs.
Sourcing slot emptyAt least one of L, B, I must be selected.
Repeated letterLL or lL makes no sense — each letter appears at most once.
ITEM not 3 charsThe abbreviation slot is exactly 3 characters; shorter or longer is malformed.
Duplicate RM codeTwo RMs cannot share the same canonical code; collision is rejected with 409.

Branded Layer Sub-Structure

When an RM has Branded sourcing, the data tree under it has 3 levels:

RM (e.g., HN-AM-P-Bl-BTR)
└── Branded layer
    ├── Brand 1 (e.g., Amul)
    │   ├── SKU 1.1 (e.g., 500G)
    │   │   ├── Supplier 1.1.1 (e.g., Sharief, V1a)
    │   │   └── Supplier 1.1.2 (e.g., Zepto, V3)
    │   └── SKU 1.2 (e.g., 100G)
    │       └── Supplier 1.2.1 …
    └── Brand 2 …

Each level is infinitely expandable — any number of brands per RM, any number of SKUs per brand, any number of suppliers per SKU.
