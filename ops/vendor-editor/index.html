📖 Vendor Code Legend

Decode any vendor code on this page without leaving.

Code Anatomy — 5 segments

{PAY_SEQ}-{SELLS}-{OPM}-{PMS}-{IDENTITY}
PAY_SEQPf / Rf / Pfr / Rfp — cash-flow direction (case-encoded primary + alt)
SELLSL / B / Lb / Bl — RM-scope stock (case-encoded primary + alt)
OPMM / A / Ma / Am — order-placement method (case-encoded primary + alt)
PMSC / B / Cb / Bc — payment-rail support (case-encoded primary + alt)
IDENTITY3-10 char uppercase abbreviation, unique across all vendors

Universal case convention (v8). All four architectural dimensions follow the same pattern: uppercase letter = primary rail/direction, lowercase letter = also possible (alternate), letter absent = impossible. The grammar fully encodes possibility and impossibility — no information thrown away.

Example walkthrough — Rf-L-M-C-PRABHU

Rf = Receive-first only (milk delivered daily, settled later — pay-first letter absent → pay-first impossible). L = loose only (branded letter absent). M = manual only (auto absent). C = cash only (bank absent). PRABHU = identity.

Full-alternates example — Rfp-Lb-Ma-Cb-EXAMPLE

A vendor where every dimension carries both rails: Receive-first primary + Pay-first alt; Loose primary + Branded alt; Manual primary + Automatable alt; Cash primary + Bank alt. Each lowercase letter declares "this is also possible" — the alt rail isn't impossible, it's just not the default.

PAY_SEQ values · cash-flow direction

CodeMeaningReal example
Pf Pay-first only. Money always leaves our account before goods/service arrives. Pure quick commerce, strict prepaid. Pf-B-A-B-ZEPTO, Pf-L-M-C-WALKIN
Rf Receive-first only. Goods always arrive before money goes out. Daily milk, post-paid utilities, pure credit-term vendors. Rf-L-M-C-PRABHU, Rf-Lb-M-Cb-SHARIFF
Pfr Pay-first primary + Receive-first alt. Default flow is pay-first, but receive-first happens occasionally (e.g. a quick-commerce vendor that sometimes ships COD). Pfr-B-A-Bc-EXAMPLEQC (hypothetical)
Rfp Receive-first primary + Pay-first alt. Default flow is receive-first, but pay-first happens occasionally (e.g. a credit-term vendor where rare items require advance payment). Rfp-Lb-Ma-Cb-SHARIFF (hypothetical)

Case convention: uppercase letter = primary direction (the dominant flow); lowercase letter = also possible (the rarer flow). The richer V-shape (V1a / V1b / V2 / V3 / V4 / V5) at the action layer captures per-transaction choreography within whichever direction the action takes.

SELLS values · what the vendor stocks (RM-scope)

CodeMeaningReal example
L Loose only. Vendor only deals in unbranded RM by weight/volume — daily milk, vegetables, mutton, fish. Rf-L-M-C-PRABHU (loose milk)
B Branded only. Vendor only stocks SKU-coded branded goods — quick commerce, B2B platforms, dedicated brand suppliers. Pf-B-A-B-ZEPTO, Rf-B-A-Bc-HYPERPURE
Lb Loose primary + Branded alt. Mixed shop where loose dominates but branded packs are also stocked as a secondary line. Rf-Lb-M-Cb-SHARIFF (loose-led departmental)
Bl Branded primary + Loose alt. Mixed shop where branded SKUs dominate but some loose staples also stocked. Rf-Bl-M-Cb-NILDIST (hypothetical brand-led distributor)

Flat LB is no longer valid (v8). The case-encoded form preserves which side dominates — information that was lost in the old alphabetical pair. S (services) and C (capex) are reserved for the bill/expense scope.

OPM values · how we order from this vendor

CodeMeaningReal example
M Manual only. Ordering always routes through a human channel — phone, WhatsApp, walk-in, or fixed delivery route. No app/API in the loop. Rf-L-M-C-PRABHU (daily route), Rf-Lb-M-Cb-SHARIFF (WhatsApp)
A Automatable only. Ordering always goes through an app, web portal, or API — pure quick commerce, B2B platforms, marketplace integrations. Pf-B-A-B-ZEPTO (app), Rf-B-A-Bc-HYPERPURE (B2B portal)
Ma Manual primary + Automatable alt. Default ordering is manual but the vendor also exposes an app/API as a backup channel. Rf-Lb-Ma-Cb-EXAMPLEHYBRID (hypothetical)
Am Automatable primary + Manual alt. Default ordering is via app/API but a human channel exists as a fallback (call vendor when API down). Pf-B-Am-B-EXAMPLEAPP (hypothetical)

The primary letter gates the agent vs. staff emission: when M or Ma, ordering routes through a staff WA prompt; when A or Am, a Cron Worker can place the order autonomously.

PMS values · payment methods this vendor accepts

CodeMeaningReal example
C Cash-only. Vendor doesn't accept bank payments — no UPI, no card, no NEFT. Settlement always physical. Rf-L-M-C-PRABHU (small loose milk vendor)
B Bank-only. Vendor doesn't accept cash — typical for app-based platforms, registered B2B suppliers, online subscriptions. UPI / NEFT / card only. Pf-B-A-B-ZEPTO, Rf-B-A-B-HYPERPURE
Cb Hybrid · Cash primary. Vendor accepts both, but cash is the default rail; bank is the alternate. Most small/local vendors who occasionally take UPI. Rf-LB-M-Cb-SHARIFF (departmental store)
Bc Hybrid · Bank primary. Vendor accepts both, but bank is the default rail; cash is the alternate. B2B platforms that allow occasional cash settlements. Rf-B-A-Bc-HYPERPURE (B2B with COD option)

Case convention. Uppercase letter = primary rail. Lowercase letter = alternate rail. Mirrors SOURCING (Lb = loose primary + branded alt) and USAGE (Pr = primary + recipe-tier secondary). Each letter (C, B) appears at most once.

Per-transaction payment method (which channel was used, exact amount split for hybrid) lives at the action layer — not the vendor row. PMS only describes what's accepted and which is primary, not what was used.

IDENTITY rules

RuleDescription
Length3-10 characters, uppercase A-Z and 0-9 only
SourceAuto-derived from vendor name (consonants stripped of vowels), owner-overridable
UniquenessGlobally unique across ALL vendors regardless of PAY_SEQ. Rf-PRABHU blocks Pf-PRABHU.
CaseAlways uppercase. Editor normalises lowercase on save.

Why minimal · the doctrine

The vendor canonical code carries only what's mathematically necessary: the cash-flow direction (Pf/Rf) and the unique identity. Everything else — which BRAND scope (NCH/HE/HN), what they SELL (loose/branded/services/consumables) — is a computed view derived from the RM ring at query time.

This obeys the COA Doctrine: properties that can be derived from upstream entity rings are never duplicated as static fields. If a vendor stops supplying NCH, the scope shrinks automatically — no manual sync required.

Computed views explained

ViewComputed how
scopeSet of brand prefixes (NCH / HE / HN) drawn from every RM that lists this vendor as a supplier
sellsSet of layer letters {L, B, I} indicating whether the vendor appears under Loose / Branded / In-house in any RM
rms_suppliedFull list of RMs supplied — array of {rm_code, layer, brand, sku, notes} tuples

Endpoint: GET /api/vendors?vendor_code=…&include=relationships

Forbidden combinations

ForbiddenWhy caught
PAY_SEQ outside {Pf, Rf, Pfr, Rfp}Enumerated set. Suffixes 'fr' and 'fp' fix the primary/alt order; Prf / Rpf rejected.
SELLS = "LB" (flat alphabetical)v8 rejects flat ordering — use Lb or Bl with a primary letter.
SELLS outside {L, B, Lb, Bl}Same case-encoded pattern. Lowercase-first (lB), duplicates (Ll), or both uppercase (LB) are rejected.
OPM outside {M, A, Ma, Am}Same case-encoded pattern.
PMS outside {C, B, Cb, Bc}Same case-encoded pattern. Lowercase-first (cB), duplicates (Cc), or both uppercase (CB) are rejected.
IDENTITY emptyIdentity required, ≥3 chars
IDENTITY too long (>10)Codes need to stay scannable
IDENTITY collisionGlobally unique — any case-insensitive duplicate is rejected with 409

All four architectural dimensions enforce the same case-encoded pattern: uppercase primary, optional lowercase alt, no duplicate letter. The grammar fully encodes possibility from impossibility — no ambiguity remains.

Connection to RM action layer

Vendor codes combine with RM action codes downstream. A purchase is encoded as {ACTION}-{RM_CODE}-{VENDOR_CODE}-…. The RM ring tells us what was bought; the vendor ring tells us who supplied, how cash flowed, what they sold, how we ordered, and which payment methods they accept.

See the COA Doctrine for the full ring composition.
