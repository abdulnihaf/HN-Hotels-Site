SECTION 1

The minimal vendor identity

The vendor canonical code carries only what's mathematically necessary: PAY_SEQ (the architectural cash-flow binary) and IDENTITY (the unique abbreviation). Everything else — which BRANDs they supply, what they SELL — is a computed view derived from the RM ring at query time.

This obeys the COA Doctrine: properties that can be derived from upstream entity rings are never duplicated as static fields. A vendor that stops supplying NCH automatically loses its NCH scope; no manual sync required.

SECTION 3

PAY_SEQ directory Position 1 · 2 chars

The architectural binary that classifies every vendor by cash-flow direction.

CodeMeaningReal examples
Pf Pay-first. Money leaves our account before goods/service arrives. Quick commerce apps, prepaid subscriptions, advance rent, prepaid utilities. Pf-ZEPTO · Pf-RENT · Pf-INTERNET
Rf Receive-first. Goods/service arrives before money goes out. Daily milk, departmental stores with credit terms, post-paid utilities, monthly bills. Rf-PRABHU · Rf-SHARIFF · Rf-ELECTRICITY
Why a binary, not a 6-letter V-shape?

The richer V-shapes (V1a / V1b / V2 / V3 / V4 / V5) capture the precise cash-flow choreography of a single purchase event. Those live at the action layer and modulate per-purchase, per-supplier within a single vendor record. PAY_SEQ at the vendor level only captures the dominant direction — that's enough for the entity ring; the action ring fills in the granularity.

SECTION 4

IDENTITY directory Position 2 · 3-10 chars

A 3-10 character uppercase abbreviation that gives the vendor its identity within the global vendor namespace.

Length

3 to 10 characters, uppercase A-Z and 0-9 only. Codes need to stay scannable.

Source

Auto-derived from vendor name (consonants, vowels stripped, deduplicated). Owner override allowed.

Uniqueness

Globally unique across ALL vendors regardless of PAY_SEQ. Rf-PRABHU blocks Pf-PRABHU.

Case

Always uppercase. Editor and API normalise lowercase on save.

SECTION 7

Worked Examples — Reading codes left to right

Six real codes, each walked through one segment at a time.

1 · Rf-PRABHU — Loose milk vendor

Rf-PRABHU │ └─ PRABHU — identity (loose buffalo milk vendor in Shivajinagar) └───── Rf — Receive-first (delivers milk daily; settled weekly/monthly)
Reads as: "A receive-first vendor named PRABHU." The computed view fills in: scope = NCH (only NCH consumes buffalo milk), sells = L (loose layer of NCH-AM-P-Lb-MLK).

2 · Pf-ZEPTO — Quick commerce app

Pf-ZEPTO │ └─ ZEPTO — identity (the quick commerce platform) └───── Pf — Pay-first (you pay in app, then goods arrive in 10 minutes)
Reads as: "A pay-first vendor named ZEPTO." Computed: scope = HN (cross-brand — both NCH and HE order from Zepto); sells = B (branded SKUs only — Zepto doesn't deal in loose).

3 · Rf-SHARIFF — Departmental store with credit terms

Rf-SHARIFF │ └─ SHARIFF — identity (Shivajinagar departmental store) └───── Rf — Receive-first (we collect goods, settle monthly bill)
Reads as: "A receive-first vendor named SHARIFF." Computed: scope = HN (cross-brand); sells = L + B (loose AND branded — they stock both formats).

4 · Pf-RENT — Landlord, advance rent

Pf-RENT │ └─ RENT — identity (landlord — functional handle, not personal name) └───── Pf — Pay-first (rent paid 1st of every month, in advance for that month)
Reads as: "A pay-first vendor named RENT." Computed: scope is empty for now (rent isn't an RM purchase — it's an opex line). The RM ring won't list this vendor; that's fine — orphan vendors are valid for non-RM expense relationships.

5 · Rf-ELECTRICITY — Post-consumption utility

Rf-ELECTRICITY │ └─ ELECTRICITY — identity (BESCOM, the Bangalore electricity board) └───── Rf — Receive-first (you consume electricity, then get the monthly bill)
Reads as: "A receive-first vendor named ELECTRICITY." Like rent — orphan in the RM sense, but real for opex tracking. PAY_SEQ correctly classifies it as receive-first (post-consumption billing).

6 · Pf-INTERNET — Subscription, advance payment

Pf-INTERNET │ └─ INTERNET — identity (BSNL or whichever ISP) └───── Pf — Pay-first (monthly fee paid in advance for the next billing period)
Reads as: "A pay-first vendor named INTERNET." Distinguishes itself from Rf-ELECTRICITY precisely because subscriptions reverse the cash-flow direction — that's the value PAY_SEQ adds.
SECTION 8

Why minimal · the doctrine

An obvious instinct would be to encode BRAND scope (NCH/HE/HN) directly in the vendor code: Rf-NCH-PRABHU. We deliberately don't, because:

  • BRAND is a relationship, not an identity. It changes when the RM ring changes — without any vendor mutation. Encoding it statically would create sync drift the day NCH stops using PRABHU.
  • SELLS is even more derived. Whether a vendor sells Loose or Branded depends on which RM layer their record sits in. That answer comes from the RM ring, not the vendor row.
  • PAY_SEQ is the only static fact. Whether you pay before or after receiving is a property of the vendor's commercial structure — it doesn't change based on which RM you bought.

So the canonical code carries only PAY_SEQ + IDENTITY. The richer view — what scope, what sells, which RMs — is computed at query time via ?include=relationships, by walking rm_sourcing_profiles.data_json for vendor refs. Ledger-style: the entity is the immutable kernel; properties bloom from the relationships.

In the static row
  • vendor_code (Pf/Rf-IDENTITY)
  • pay_seq, identity_abbr, vendor_name
  • communication / location / payment / GST in data_json
Never in the static row
  • BRAND scope
  • SELLS categories
  • RMs supplied
  • V-shape (lives at action layer)
SECTION 9

Connection to the RM ring + Action layer

The vendor ring is the second entity ring, sitting alongside the RM ring. Both feed downstream into the Action ring (purchases, productions, transfers).

Composition of an Action code (sketch) 
{ACTION}-{RM_CODE}-{VENDOR_CODE}-{V_SHAPE}-{TXN_ID}

Examples:
  PUR-NCH-AM-P-Lb-MLK-Rf-PRABHU-V1a-2026050801   ← daily milk receipt
  PUR-HN-AM-P-Bl-BTR-Pf-ZEPTO-V3-2026050802     ← Amul butter via Zepto
  EXP-OPEX-Pf-RENT-V5-2026050801                ← monthly rent payment
RM ring

Five segments — what was bought. Vendor refs live inside data_json at the L / B / I layers.

Vendor ring (this page)

Two segments — who supplied + cash-flow direction. Computed views fill in scope/sells via the RM ring.

Action ring

Composes RM + Vendor + V-shape into a transactional code that an Event can aggregate. Defined in COA Doctrine.

Two entity rings, one action ring, one event ring. Every action code is a coordinate; every event aggregates actions. The vendor ring stays minimal so the action ring stays composable.
