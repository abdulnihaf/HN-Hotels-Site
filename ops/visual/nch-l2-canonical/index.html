The L2 Action Code Grammar

A single grammar that captures every purchase or production event. The code itself encodes the intelligence — agents downstream (L3 payment, L4 inventory, Price-Intel) read the code, never reality.

{RM_CODE} / {LAYER} / {V_SHAPE}-{SUPPLIER}[-{BRAND-SKU}]
RM_CODE
NCH-AM-MLK
Identity of what's being sourced. Already locked in Section 2 of foundation. Read by L4 inventory.
LAYER
L · B · I
L = Loose vendor. B = Branded SKU. I = In-house produced. Read by Price-Intel for research stack.
V_SHAPE
V1 · V2 · V3 · V4 · V5
Payment behaviour. Already locked in Section 4. Read by L3 to fire payment at the right moment.
SUPPLIER
PRABHU · ZEPTO · …
Vendor abbreviation. Cleaned, uppercased. For I-layer this slot becomes the recipe identifier.
BRAND-SKU
AMUL-5L · TATA-1KG
Only present when LAYER=B. Brand abbreviation + pack size. Lets Price-Intel scrape exact SKU on Hyperpure / Blinkit / Zepto.

The Three Sourcing Layers · L · B · I

L · LOOSE vendor-direct
Bought from a vendor in unbranded form. No SKU. Quantity in standard units (kg, L, pcs).
NCH-AM-MLK / L / V1-PRABHU
Examples:
  • Buffalo Milk from Prabhu (₹55/L)
  • Loose tea powder from Afeefa
  • Buns from Sangeetha bakery (V2 credit)
  • Samosa raw from Sameer Hamza
Price-Intel research: B2B portals (IndiaMart, mandi rates, wholesale catalogs)
B · BRANDED defined SKU
A specific brand × specific pack-size SKU. Bought from either a vendor who stocks it, or quick-commerce platform.
NCH-AM-MLK / B / V3-ZEPTO-AMUL-1L
Examples:
  • Amul Butter 500g from Zepto
  • Tata Tea Premium 1kg from Hyperpure
  • Bisleri 500ml × case from BigBasket
  • Same SKU, different supplier = different code
Price-Intel research: D2C / Quick-Commerce (Hyperpure, Blinkit, Zepto, BigBasket)
I · IN-HOUSE recipe-produced
Produced internally from upstream RMs through a recipe. No vendor — the kitchen itself is the source.
NCH-AS-BUN / I / R-MDA+YST+BTR+SGR
Examples (current + future):
  • Milk batch (80L raw → 75L boiled chai-base) — already live
  • Decoction (tea + spices → 9L) — already live
  • Bun bake (future): maida + yeast + butter + sugar
  • Biscuit bake (future): maida + ghee + sugar + spice
L4 effect: consumes upstream RMs, produces output RM

Scenario Walkthroughs · Same RM, Different Operational Realities → Different Action Codes

Each scenario below is a real or hypothetical NCH operational reality. The architecture stays identical — only the code changes.

Scenario 1 · Buffalo Milk has 4 sourcing realities (one RM, four codes)
LV1today's primary
NCH-AM-MLK / L / V1-PRABHU
Daily route delivery from Prabhu. ₹55/L. Cash settles same day. 51 POs in 180 days.
LV4emergency backup
NCH-AM-MLK / L / V4-RPRASAD
Walk-in cash from R Prasad when Prabhu can't supply. ₹80/L (premium). 1 PO observed.
BV3future · quick commerce
NCH-AM-MLK / B / V3-ZEPTO-AMUL-1L
Amul 1L pouch ordered on Zepto. Payment fires before delivery. Different sequence to L1/V4.
BV1future · vendor SKU
NCH-AM-MLK / B / V1-SHARIFF-AMUL-5L
Same Amul, different SKU + different supplier. Sharief stocks both loose milk and branded pouches.
⇢ One RM. Four codes. Four different L3 payment flows. Two different price-intel research stacks.
Scenario 2 · Buy a bun vs bake a bun (the in-house question)
LV2today · direct buy on credit
NCH-AS-BUN / L / V2-SANGEETHA
Buns delivered daily by Sangeetha bakery. Bills monthly. Pay-on-due. PO → Receive → Pay sequence.
Ifuture · in-house bake
NCH-AS-BUN / I / R-MDA+YST+BTR+SGR
Recipe consumes maida + yeast + butter + sugar from stock. No vendor, no V-shape. L3 untouched. L4 sees both consumption (upstream) and production (output).
Architectural insight: The same RM (NCH-AS-BUN) can have its STOCK created by either a purchase (L/B-layer) or a production event (I-layer). Both events end with bun in inventory. L3 fires only for purchases. L4 reconciles both.

Anomaly Guards · Impossible Combinations the System Catches

L + V3 = ERROR
NCH-AM-MLK / L / V3-ZEPTO
Quick-commerce only sells defined SKUs. There is no "loose" on Zepto. Code = error, must be B.
B without BRAND-SKU
NCH-AM-MLK / B / V3-ZEPTO
Branded purchases must carry the SKU identifier. Missing SKU = malformed code.
L with BRAND-SKU
NCH-AM-MLK / L / V1-PRABHU-AMUL-1L
Loose has no SKU by definition. SKU present means the layer is wrong — should be B.
I without recipe
NCH-AS-BUN / I / —
In-house must reference a recipe (R-…). Missing = system can't compute upstream consumption.
V5 with PO record
NCH-AM-WTR / L / V5-GOLDENRAJESH + PO
Standing invoices have no PO step. PO present means it should be V2 instead.
RM_CODE not in registry
NCH-XX-??? / L / V1-PRABHU
RM must exist in the 37 canonical RM list. New RM = registry update first, then action code.

Why This Architecture Holds

No reality interpretation needed.
Agents read codes. Codes carry intelligence. The system never has to "look at" the real-world operation — the code already encodes it.
Tomorrow's scenarios = today's grammar.
Adding Hyperpure, in-house bakery, second outlet, new credit vendor, new branded SKU — none change the architecture. Just new instances of the same grammar.
Errors visible at the code layer.
Impossible combinations (L+V3, B without SKU, I without recipe) fail validation at code-creation, before they pollute downstream agents.
