Architectural Deltas vs NCH-L1 · Same skeleton, simpler mechanic

Identity
NCH: RNR_64–RNR_68 (5 fixed runners with barcodes + dedicated QRs)
HE: POS_USER_PIN — whoever logs into the POS to take the order owns settlement responsibility for that ticket. Flexible, anyone can be on shift.
Captain promise
NCH runner: owes cash AND UPI (per-runner Razorpay QR fan-in)
HE captain/waiter: owes only cash. UPI flows direct to the bank — no per-captain QR. Settlement equation is pure cash. Simpler.
UPI gateway
NCH: Razorpay live API + webhook (✓ events real-time)
HE: Paytm (no API key yet) → manual CSV reconciliation by agent until API arrives. Lower volume = no live data isn't a deal-breaker.

Equations · The Mathematical Spine

EQ_HE_CAPTAIN_CASH_OWED(USR_PIN_X) =
  Σ pos.payment WHERE
    config_id ∈ {POS_5, POS_6, POS_32}
    AND pm_id ∈ {PM_11, PM_19, PM_64}
    AND user_id_pin = PIN_X
  ⇒ MUST = cash returned at shift close per PIN
EQ_HE_PAYTM_RECON =
  Σ pos.payment WHERE pm_id ∈ {PM_14, PM_52, PM_58}
  vs Paytm CSV (manual upload pending API key)
  ⇒ drift > ₹500 fires RULE_H_PAYTM_DRIFT
EQ_HE_AGG_NET_EXPECTED(platform) =
  Σ POS_7_gross × (1 − comm_pct ± 0.5%)
  ⇒ match BNK_HDFC_HE credit T + contract days
EQ_HE_CASH_DRAWER =
  opening_float
  + Σ cash_in (PM_11 + captain settled + waiter settled)
  − Σ cash_out (counter expenses + handovers_out)
  ⇒ MUST = physical drawer count
EQ_LIVE_INVENTORY(item) = // shared with NCH
  closing_count(yesterday)
  + Σ purchase.receive(item, today)
  − Σ pos.sale × BoM(item, product)
  − Σ wastage(item, today)
