The Two-Batch Spine — NCH operational logic

NCH is fundamentally a two-batch business. The decoction pool and the milk pool are the upstream RM consumers; ~80% of POS items pull a portion from one or both. The remaining ~20% are direct-from-shelf (resale: biscuits, water, niloufer boxes) or simple def+apx wraps (bun + spread). The Khajoor batch is a third, smaller batch that produces 155 pieces per run.

Milk Batch 75 L yield · 93.75%
80 L Buffalo Milk + 2 kg SMP + 1.5 kg Sugar + 1.5 kg Condensed Milk
→ 3-hr reduction → 75 L Milk Batch
Variance band: 5–10% on yield (cook drift) and 5% on per-cup dispense (60ml/cup target).
Decoction Batch 3.6 L · ~90 servings
9 L Filter Water + 400 g Tea Powder + 800 g Sugar
→ 1-hr boil on bhati (4 kg charcoal) → 3.6 L Decoction
Variance band: 5–10% on yield, 5% on per-cup dispense (40ml/cup standard, 60ml for Kadak).
Khajoor Batch 155 pcs / run
5 kg Maida + 2 kg Sooji + 200 g Custard Powder + 100 g Saunf + 6 L Oil
+ 200 g Kaju + 200 g Pumpkin Seeds + 30 g Baking Soda + 50 g Elaichi
→ 155 pieces
Variance band: 5% on piece-count yield. Once baked, each Khajor sold = 1 DEF.

L2 stack · custom build target (independent of L3)

The L2 layer — RM master, recipes (10 shapes), batch runs, per-sale RM consumption, variance scoring — moves to D1 + Cloudflare Functions. L3 (vendor bills, salary, expense) lives in its own parallel D1 schema; the two cross-reference only at month-close on the L2 → L3 closure point. Odoo (odoo.hnhotels.in) retreats from operational truth to nightly compliance posting. POS Odoos (test/ops.hamzahotel.com) stay untouched — they remain L1's sales-truth source.

D1 schema sketch · 6 tables drive the entire layer
-- 1. Canonical RM master (replaces Odoo product.template + product.category)
CREATE TABLE rm_products (
  id            INTEGER PRIMARY KEY,
  code          TEXT UNIQUE,         -- HN-RM-XXX or NCH-XX
  name          TEXT,
  category      TEXT,
  uom           TEXT,                -- kg, g, L, ml, Units, can ...
  current_cost_paise INTEGER,        -- live vendor price
  vendor_id     INTEGER,
  is_batch_intermediate BOOLEAN      -- true for "Milk Batch", "Decoction", "Khajoor Batch"
);

-- 2. Per-RM tolerance bands (variance scoring)
CREATE TABLE rm_tolerance (
  rm_id         INTEGER REFERENCES rm_products(id),
  tolerance_pct INTEGER            -- 0 for DEF, 5–15 for APX
);

-- 3. POS recipe = which shape + parameters
CREATE TABLE pos_recipe (
  pos_item_id   INTEGER PRIMARY KEY,  -- maps to product_id from POS Odoo
  brand         TEXT,                -- 'NCH' or 'HE'
  shape_code    TEXT,                -- 'S1'..'S10'
  params_json   TEXT                 -- shape-specific component list
);

-- 4. Production batch run (NEW concept — doesn't exist in Odoo today)
CREATE TABLE batch_run (
  id            INTEGER PRIMARY KEY,
  batch_rm_id   INTEGER REFERENCES rm_products(id),
  brand         TEXT,
  started_at    TEXT,
  completed_at  TEXT,
  expected_yield REAL,               -- 75 L for Milk, 3.6 L for Decoction
  actual_yield  REAL,                -- staff-logged at end of cook
  prepared_by_pin TEXT
);

-- 5. Inputs to a batch run (denorm of recipe at run time)
CREATE TABLE batch_run_input (
  batch_run_id  INTEGER REFERENCES batch_run(id),
  rm_id         INTEGER REFERENCES rm_products(id),
  qty_used      REAL,
  uom           TEXT
);

-- 6. Per-sale RM consumption (one row per recipe component per sale)
CREATE TABLE rm_consumption_event (
  id            INTEGER PRIMARY KEY,
  pos_order_line_id TEXT,           -- from POS Odoo
  occurred_at   TEXT,
  rm_id         INTEGER,
  source_batch_run_id INTEGER,      -- NULL if direct-RM (no batch)
  expected_qty  REAL,                -- from recipe
  actual_qty    REAL,                -- system-deducted = expected (no live measure)
  variance_qty  REAL                 -- computed at batch close-out
);
Daily flow · how variance gets caught at the moment of error
  1. Batch start (e.g., 6:30am): Mujib opens "Start Decoction Batch" on PIN-gated UI. System creates batch_run with expected_yield = 3.6 L, decrements rm_products on-hand for 9 L water + 400g tea + 800g sugar (the inputs).
  2. Batch end (~7:30am): Mujib enters actual yield (e.g., "3.4 L"). System computes batch yield variance = (3.4 − 3.6) / 3.6 = −5.5% — within tolerance. Logged to batch_run.actual_yield.
  3. Throughout the day: Each Irani Chai sale at the POS triggers rm_consumption_event rows: 60ml from Milk Batch + 40ml from Decoction Batch. The batch's "remaining" auto-decrements.
  4. Batch close-out (when remaining = 0): System checks: did the actual servings × per-cup ml match the actual yield? If servings × 40ml > actual yield + tolerance, that's over-dispensing (free chai → leakage). If < actual yield − tolerance, under-dispensing or unrecorded sale (cash in pocket).
  5. Variance alert: Agent fires WABA to Mujib first ("Decoction batch closed with -8% variance: 80 servings sold but yield 3.4L = 85 servings expected"). 30 min SLA → escalate to owner.
Why L2 leaves Odoo (L2-specific reasons)
  • No mrp.bom on this Odoo — manufacturing module not installed → can't model the two-layer BoM (Decoction batch + per-cup portion) at all.
  • No first-class concept of "batch run" — Odoo has no native "Mujib started a 9L decoction batch at 6:30am, finished at 7:30am with 3.4L actual yield" event. We'd have to misuse mrp.production or invent custom models — both fragile.
  • Variance & WABA-alert agents need millisecond-grade reads from a local D1 — not JSON-RPC round-trips to a hosted Odoo. Per-sale recipe deduction at POS scale needs < 50ms; Odoo can't.
  • 10-shape recipe vocabulary doesn't fit Odoo's BoM model cleanly. Custom pos_recipe table with shape_code + params_json is dead simple in D1.
  • Cost-of-sale crosses into L3 at month-close, not continuously. Keeping L2 in D1 means the L1↔L2 closure is fast and L2↔L3 is a once-a-month batch — no live coupling needed.
  • POS Odoos (HE prod, NCH prod) remain untouched as L1's source. Compliance Odoo (odoo.hnhotels.in) keeps GST/audit ledger only.
