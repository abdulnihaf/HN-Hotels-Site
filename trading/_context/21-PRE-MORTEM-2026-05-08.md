## 3 · Real-money path — failure modes I walked through

`place_bracket` flow (functions/api/kite.js lines 519-688):

```
1. Validate body (exchange, tradingsymbol, quantity, stop_price, target_price)
2. Check user_config.block_real_orders — refuse if "1" with simulated response
3. Insert tracking row in kite_bracket_orders
4. STEP 1: POST /orders/regular MARKET BUY
5. STEP 2: poll /orders/{id} for COMPLETE status — max 12 seconds (12 × 1s)
6. STEP 3: POST /gtt/triggers OCO (stop+target legs) — up to 3 retries
7. STEP 4: if GTT fails, fallback to single-leg SL-M (stop only, no target)
```

### 3.1 Failure mode tree

| # | Scenario | Detection | Recovery |
|---|---|---|---|
| **F1** | block_real_orders='1' (correctly set, you forgot to flip) | Response: `{ok:false, blocked:true, reason:'shadow_run', simulated:{...}}` | Run SQL: `UPDATE user_config SET config_value='0' WHERE config_key='block_real_orders';` ONLY IF Kite has funds (check §2.F) |
| **F2** | Kite rejects MARKET BUY (insufficient margin / circuit / freeze) | Response steps[0].step='place_buy', ok=false. result.error contains Kite message | Read kite_api_log for the exact message. Common: "INSUFFICIENT FUNDS" → top up Kite. "ORDER FREEZE" → stock in upper/lower circuit, pick another. |
| **F3** | BUY placed but fill timeout (12s elapsed, no COMPLETE) | Response: steps include {step:'fill_polling', ok:false, error:'fill_timeout'}. **Order is STILL ALIVE in Kite — may fill later** | Check `/api/kite?action=order_status` with the order_id. If still pending: cancel it `/api/kite?action=cancel_order` to avoid surprise fills. If filled in the meantime: position is unmanaged — **manually place GTT in Kite app**. |
| **F4** | Partial fill ignored (e.g. 50 of 80 shares filled) | The poll only checks `status=='COMPLETE'`. **Partial fills DON'T trigger this branch.** Reported as fill_timeout. | Same as F3 — check Kite app directly. If partial filled: need to either place GTT for the partial OR cancel remainder + place GTT for filled qty. **Manual intervention required.** |
| **F5** | BUY succeeds, GTT all 3 retries fail | Response: steps include {step:'gtt_failed', ok:false}, then {step:'fallback_sl_m'} | Falls back to SL-M (stop only). Position has stop but **NO TARGET**. Manually set target via Kite app. |
| **F6** | GTT fails AND fallback SL-M also fails | Response.warning = "POSITION HAS NO STOP. Place stop manually in Kite immediately." | **CRITICAL: position is naked**. Place SL-M manually in Kite within 30 seconds. |
| **F7** | Network timeout to Kite during BUY POST | kiteFetch returns `{ok:false, _err:'...'}`. **Cannot tell if Kite received the order or not.** | Wait 5s, then `/api/kite?action=orders` to list today's orders. If your tag (HN_WE_BRK) appears: order placed, proceed. If not: safe to retry. |
| **F8** | Kite token expires between BUY (succeeds) and GTT (fails with auth) | GTT all 3 retries fail with 401. Fallback SL-M also fails with 401. F6 scenario. | Re-OAuth at /wealth/auth/login. Then manually place GTT in Kite app for the existing position. |

### 3.2 Pre-existing risk flags I want you aware of

- **product='CNC' is the default** in place_bracket (line 562). For intraday MIS you must pass `product:'MIS'` in the body. Otherwise the order goes through as DELIVERY which has very different margin requirements + can't be intraday-squared. **Verify the /execute/ confirm modal sets product='MIS'**.
- **stop_price + target_price are in rupees** (parsed via `parseFloat`), not paise. The /execute/ frontend must convert paise→rupees correctly before POST. Easy to swap units.
- **No idempotency key** on the place_bracket request. If owner double-taps confirm OR Cloudflare retries, you could place 2 BUY orders. Kite's `tag` field is descriptive only, not deduplication.
